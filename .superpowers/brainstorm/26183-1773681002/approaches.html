Three Architecture Approaches

How should Manna Ray's plugin be structured?

A

Recommended: Thin Plugin + Skill Registry

A lightweight Claude Code plugin that provides:

  • Skill Registry — loads skill prompts from .md files, registers as slash commands
  • Context Manager — init/validate/load context files, inject into skill prompts
  • Pipeline Runner — defines workflows as ordered skill sequences with dependency checks
  • State Store — simple JSON tracking what's been run, outputs produced, workflow progress

Pros

  • Skills stay as .md files — easy to edit/iterate prompts
  • Minimal code surface — faster to build
  • Plugin handles plumbing only
  • New skills = add a .md file + register

Cons

  • Less control over skill execution
  • Output format depends on Claude following prompts
  • Harder to enforce strict contracts between skills
B

Schema-Enforced Plugin

Same plugin structure, but each skill gets a TypeScript schema defining:

  • Input contract — what context files and prior outputs are required
  • Output contract — JSON schema for what the skill must produce
  • Validation layer — code that checks inputs before running, validates outputs after
  • Transform layer — code that extracts structured data from Claude's response for downstream skills

Pros

  • Reliable skill chaining — outputs guaranteed to match next skill's inputs
  • Catches errors early (missing prerequisites)
  • Enables "dry run" to preview what a workflow needs
  • Structured outputs enable dashboards/reporting later

Cons

  • More upfront engineering per skill
  • Schema maintenance overhead
  • Risk of over-constraining Claude's flexible reasoning
C

Full Application Layer

Manna Ray as a substantial TypeScript application:

  • Own database — SQLite for state, history, outputs, relationships
  • Structured API calls — skills call Claude API directly with structured tool use
  • Rich CLI UI — Ink/React-based terminal UI with progress, menus, dashboards
  • Plugin integration — exposes results back to Claude Code via slash commands

Pros

  • Maximum control and reliability
  • Best UX with rich terminal interface
  • Could become standalone product later
  • Full history and analytics

Cons

  • 10x engineering effort
  • Loses Claude Code's conversational flexibility
  • Maintaining own API integration is complex
  • Premature optimization — validate the concept first

Recommendation: Start with A, evolve toward B

Build the thin plugin first to validate the concept quickly. As you learn which skill boundaries need enforcement, add schemas incrementally where they matter (workflow handoff points). Avoid C until the product is validated — it's a rebuild, not an evolution.
