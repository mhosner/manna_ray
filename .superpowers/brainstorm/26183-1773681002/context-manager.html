Section 3: How context files are initialized, loaded, validated, and kept fresh

Context File Lifecycle
/manna context init
━━━━━━━━━━━━━━━━━━━
  Scaffolds context/ directory with template files:

  context/
  ├─ product.md      ← roadmap, metrics, known issues, positioning
  ├─ company.md      ← strategic priorities, team structure, biz model
  ├─ personas.md     ← user archetypes, JTBD, pain points
  ├─ competitors.md  ← competitive intel, win/loss, pricing
  └─ goals.md        ← quarterly/annual OKRs, success metrics

  Each file gets a structured template with sections to fill.
  Claude guides the PM through populating them interactively.


Before each skill runs:
━━━━━━━━━━━━━━━━━━━━━━
  1. DEPENDENCY CHECK
     Skill declares which context files it needs
     e.g., prd-generator requires: [product.md, personas.md, goals.md]

     Missing?  → "product.md not found. Run /manna context init
                  or create it manually."
     Empty?    → "personas.md exists but has no content.
                  Want to populate it now?"

  2. STALENESS CHECK
     State store tracks last-modified checksums per context file.
     If a file hasn't changed in 30+ days:

     "⚠ competitors.md hasn't been updated in 47 days.
      Want to review it before running landscape-mapper?"

  3. CONTEXT INJECTION
     Plugin reads required context files and prepends them to
     the skill prompt:

     ┌─────────────────────────────────────┐
     │  ## Current Context                 │
     │  ### Product (from product.md)      │
     │  [contents]                         │
     │  ### Personas (from personas.md)    │
     │  [contents]                         │
     │                                     │
     │  ## Skill Instructions              │
     │  [SKILL.md contents]               │
     │                                     │
     │  ## User Input                      │
     │  [what the PM typed]               │
     └─────────────────────────────────────┘


After skill completes:
━━━━━━━━━━━━━━━━━━━━━━
  Skills that produce new insights trigger context update prompts:

  SKILL                        SUGGESTS UPDATING
  ─────────────────────────    ──────────────────
  user-interview-analyzer   →  personas.md
  jtbd-extractor            →  personas.md
  competitive-profile       →  competitors.md
  landscape-mapper          →  competitors.md
  north-star-finder         →  goals.md, product.md
  ab-test-analyzer          →  product.md
  funnel-analyzer           →  product.md
  quarterly-planning        →  goals.md
  pricing-strategy          →  product.md, competitors.md
  positioning-statement     →  product.md

  The plugin asks:
  "This analysis produced new insights about [topic].
   Want to update [file] with these findings?"

  If yes → Claude drafts the update, PM approves,
           state store records new checksum.

Design Principles

Declare, don't discover

Each skill's .md file declares its context dependencies in frontmatter. No guessing, no runtime scanning. Missing a dependency? Fail fast with a clear message.

Suggest, don't force

Staleness warnings and update prompts are suggestions. The PM can skip them. Context files are the PM's responsibility — the plugin just makes maintenance easier.

Outputs feed context

The feedback loop isn't just between workflows — it's between skill outputs and context files. Every insight should have a path back into the shared knowledge base.

Incremental population

Context files don't need to be complete on day one. Skills work with what exists and flag what's missing. The knowledge base grows organically as you use the system.
